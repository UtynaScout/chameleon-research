# DPI Resistance Research вҖ” Phase 6

> **Ethical Disclaimer:** These features are designed exclusively for
> research into DPI detection systems in controlled lab environments.
> They are NOT intended for circumventing lawful network restrictions.

## Overview

Phase 6 adds configurable DPI-resistance features to the NetSynth VPN
to study how Deep Packet Inspection systems identify tunnel traffic.
All features are opt-in via TOML configuration.

## Features

### 1. SNI Injection

**Problem:** The original VPN sent an empty SNI in the TLS Client Hello,
which DPI systems trivially flag вҖ” real browsers always include an SNI.

**Solution:** Configurable `tls_sni` field injects a domain name into the
QUIC/TLS Client Hello.

```toml
# configs/vpn-client.toml
tls_sni = "www.google.com"
```

**Implementation:** The SNI is passed as the `server_name` parameter to
`quinn::Endpoint::connect()`. Since we use an `InsecureServerVerifier`,
the SNI doesn't need to match the server certificate.

**Files:** `examples/vpn-client.rs`, `src/transport/quic.rs`

**Verification:**
```bash
tshark -r capture.pcap -Y "tls.handshake.type == 1" \
    -T fields -e tls.handshake.extensions_server_name
```

### 2. TLS Fingerprint Presets

**Problem:** Different TLS implementations advertise cipher suites in
different orders. The JA3 fingerprint of rustls is unique and easily
identifiable by DPI.

**Solution:** Three configurable presets that reorder cipher suites to
match popular browsers:

| Preset | First TLS 1.3 Suite | ChaCha Position |
|--------|-------------------|-----------------|
| `rustls_default` | AES-256-GCM (0x1302) | Last |
| `chrome_130` | AES-128-GCM (0x1301) | After AES-256 |
| `firefox_120` | AES-128-GCM (0x1301) | Before AES-256 |

```toml
tls_fingerprint = "chrome_130"
```

**Implementation:** Custom `rustls::crypto::CryptoProvider` instances
with reordered `cipher_suites` and `kx_groups` vectors, applied via
`ClientConfig::builder_with_provider()`.

**Files:** `src/transport/dpi.rs`, `src/transport/handshake.rs`

**Limitations:**
- Extensions order is controlled by rustls internals
- Some browser-specific extensions (e.g., GREASE, PSK) are not mimicked
- The full JA4 fingerprint includes more fields than cipher suites

### 3. Packet Padding

**Problem:** VPN tunnels produce characteristic packet size distributions
(peaks at 84, 152, 81, 265 bytes in our original traffic). Statistical
analysis of these distributions can identify VPN traffic.

**Solution:** Configurable padding modes that normalize packet sizes:

| Mode | Description |
|------|-------------|
| `none` | No padding (backward compatible) |
| `mss` | Pad to nearest MSS value (e.g., 1200, 1350, 1500) |
| `fixed` | Pad all packets to exact size |
| `random` | Pad to random size in [min, max] range |

```toml
[padding]
enabled = true
mode = "mss"
mss_values = [1200, 1350, 1500]
```

**Wire format with padding:**
```
[2B ciphertext length BE][encrypted([2B original_len BE][data][random_padding])]
```

When `original_len == 0`, the packet is a dummy/keepalive (discarded by
receiver).

**Files:** `src/transport/dpi.rs`, `src/tun/mod.rs`

**Both client and server must use matching padding configuration.**

### 4. Traffic Shaping

**Problem:** VPN traffic has characteristic timing patterns (uniform
inter-packet delays) that differ from real application traffic.

**Solution:** A `TrafficShaper` wrapper that re-schedules packet
transmission according to statistical profiles generated by the
Weaver engine:

| Profile | Description |
|---------|-------------|
| `none` | No shaping (immediate send) |
| `browsing` | Mimics web browsing (burst-idle-burst) |
| `streaming` | Mimics video streaming (continuous with high StreamвҶ’Stream probability) |

```toml
shaping = "browsing"
```

**Implementation:** Packets are buffered in a queue. A background tokio
task drains the queue at intervals drawn from the Weaver engine's Markov
chain model. In idle periods where the profile indicates traffic should
exist, dummy packets are injected (requires padding to be enabled).

**Files:** `src/transport/shaper.rs`, `examples/vpn-client.rs`

## Configuration Reference

### Client (`configs/vpn-client.toml`)

```toml
# SNI for TLS Client Hello
tls_sni = "www.google.com"

# ALPN protocol
tls_alpn = "h3"

# Fingerprint preset: rustls_default, chrome_130, firefox_120
tls_fingerprint = "chrome_130"

# Traffic shaping: none, browsing, streaming
shaping = "browsing"

# Padding (must match server)
[padding]
enabled = true
mode = "mss"
mss_values = [1200, 1350, 1500]
```

### Server (`configs/vpn-server.toml`)

```toml
# Padding (must match client)
[padding]
enabled = true
mode = "mss"
mss_values = [1200, 1350, 1500]
```

## Verification Commands

```bash
# Check SNI in captured traffic
tshark -r capture.pcap -Y "tls.handshake.type == 1" \
    -T fields -e tls.handshake.extensions_server_name

# Check cipher suites in Client Hello
tshark -r capture.pcap -Y "tls.handshake.type == 1" \
    -T fields -e tls.handshake.ciphersuite

# Packet size distribution (should show MSS values, not original peaks)
tshark -r capture.pcap -T fields -e frame.len | \
    sort | uniq -c | sort -rn | head -20

# Inter-packet timing analysis
tshark -r capture.pcap -T fields -e frame.time_delta | head -30
```

| Transported over | TCP+TLS 1.3 | QUIC | QUIC вң… |

## Known Limitations

1. **TLS Extensions:** rustls controls extension ordering internally;
   only cipher suites and kx groups are configurable.
2. **QUIC Version:** No QUIC version negotiation mimicry.
3. **Padding Overhead:** MSS padding increases bandwidth ~2-5x for small
   packets.
4. **Shaping Latency:** Traffic shaping adds 1-25ms delay per packet.
5. **No GREASE:** Chrome's GREASE (random extensions) is not mimicked.
